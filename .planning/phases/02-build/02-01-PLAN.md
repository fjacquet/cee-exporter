```
BINARY_NAME    := cee-exporter
BINARY_WINDOWS := cee-exporter.exe
CMD_PATH       := ./cmd/cee-exporter
LDFLAGS        := -s -w

.PHONY: build build-windows test lint clean

build:
 CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
   go build -trimpath -ldflags="$(LDFLAGS)" -o $(BINARY_NAME) $(CMD_PATH)

build-windows:
 CGO_ENABLED=0 GOOS=windows GOARCH=amd64 \
   go build -trimpath -ldflags="$(LDFLAGS)" -o $(BINARY_WINDOWS) $(CMD_PATH)

test:
 go test ./...

lint:
 go vet ./...

clean:
 rm -f $(BINARY_NAME) $(BINARY_WINDOWS)
```

Critical implementation notes:

- Recipes MUST be indented with a literal tab character (not spaces). Use the Write tool directly â€” do NOT pipe through a shell heredoc, as editors and shells sometimes convert tabs to spaces.
- `CGO_ENABLED=0` is required on BOTH the Linux and Windows targets. The project is pure Go (only `github.com/BurntSushi/toml` and `golang.org/x/sys` â€” both CGO-free), but CGO=0 ensures the Linux binary is statically linked (no libc dependency) and the Windows cross-compile is unambiguous.
- `GOOS=linux GOARCH=amd64` is hardcoded in the `build` target (not the host arch). This matches the BUILD-01 success criterion: "Linux `cee-exporter` binary that runs and exits cleanly". Developers can use `go run ./cmd/cee-exporter` for native local testing.
- `-trimpath` removes absolute source paths from the binary (reproducible builds).
- `-ldflags="-s -w"` strips DWARF debug info and the symbol table, shrinking the binary.
- `make test` does NOT add `-race` because -race requires CGO; keeping it consistent with CGO=0 build posture.
- `make lint` runs `go vet ./...` only (not golangci-lint) â€” per BUILD-01 requirement language.
- Build package path is `./cmd/cee-exporter` (no trailing slash) â€” Go toolchain behavior with trailing slashes is undefined.
- Both `$(BINARY_NAME)` and `$(BINARY_WINDOWS)` are listed in the `clean` target so CI environments stay tidy.
