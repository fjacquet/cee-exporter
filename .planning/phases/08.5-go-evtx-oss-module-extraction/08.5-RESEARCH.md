## Summary

Phase 8.5 is a pure extraction refactoring: no new behaviour is added. The goal is to move the EVTX binary-format implementation out of `cee-exporter/pkg/evtx/` into a standalone, independently releasable Go module (`github.com/fjacquet/go-evtx`), then wire `cee-exporter` back to it as a regular `go.mod` dependency.

The source files to extract are precisely identified from the current codebase: `evtx_binformat.go` (platform-agnostic format helpers with no build tag), the BinXML encoding logic inside `writer_evtx_notwindows.go`, and the corresponding test files. The `WindowsEvent` struct stays in `cee-exporter`; the new module receives a simpler `map[string]string` field API that removes the cross-module struct coupling. The adapter in `cee-exporter` translates `WindowsEvent` → `map[string]string` before calling `go-evtx`.

Publishing to pkg.go.dev is automatic: a semver git tag (`v0.1.0`) pushed to a public GitHub repo triggers the proxy crawler. No manual approval step is needed.

**Primary recommendation:** Extract `evtx_binformat.go` and BinXML encoding helpers verbatim into `go-evtx`; expose two public functions (`WriteRaw`, `WriteRecord`); thin-adapter `BinaryEvtxWriter` translates `WindowsEvent` to `map[string]string` and delegates. Do not change any EVTX format logic during extraction — that is Phase 10's job.

### Key coupling to break

`collectSubstitutions(e WindowsEvent)` currently takes a `WindowsEvent`. In go-evtx the high-level API is:

```go
WriteRecord(eventID int, fields map[string]string) error
```

The adapter in cee-exporter constructs the `map[string]string` from a `WindowsEvent` before calling `WriteRecord`. The map keys match the current `dataFields` names in `collectSubstitutions`.

## Standard Stack

### Core

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| Go stdlib | go 1.24 | All EVTX logic, file I/O, encoding | No external dependencies: CGO_ENABLED=0 constraint; stdlib covers CRC32, UTF-16, binary LE, OS file ops |
| `hash/crc32` | stdlib | IEEE CRC for file/chunk headers | Only valid CRC algorithm per EVTX spec |
| `encoding/binary` | stdlib | Little-endian serialization | Required for all EVTX binary fields |
| `unicode/utf16` | stdlib | BinXML string encoding | EVTX requires UTF-16LE for all string values |

| Tool | Version | Purpose | Why Standard |

| `actions/setup-go` | v5 | Go toolchain in GitHub Actions | Official GitHub Action for Go |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `map[string]string` field API | Exported `EvtxRecord` struct | Struct avoids stringly-typed keys but creates shared type coupling between cee-exporter and go-evtx; map is simpler for the extraction step |

| v1 tag immediately | v0.1.0 tag first | v0.x allows breaking API changes without a v2 major bump; correct choice for a new library |

**No build tags needed in go-evtx.** The module is intentionally platform-agnostic (pure Go, no OS calls beyond `os.WriteFile`). Build-tag complexity stays in cee-exporter.

### Pattern 1: Layered Public API

**What:** Two public entry points with different abstraction levels.
**When to use:** WriteRaw for forensics tools and callers that already have a BinXML-encoded chunk. WriteRecord for application-level callers that have structured event data.

```go
// Source: API design derived from EXT-02 / EXT-03 requirements

package evtx

import "os"

// Writer manages a single .evtx output file.
type Writer struct {
    path     string
    records  []byte
    recordID uint64
    firstID  uint64
}

// New creates a Writer that will produce an EVTX file at path.
// The parent directory is created if absent.
func New(path string) (*Writer, error) { ... }

// WriteRaw appends a pre-encoded BinXML chunk to the pending buffer.
// Callers are responsible for correct BinXML encoding.
func (w *Writer) WriteRaw(chunk []byte) error { ... }

// WriteRecord encodes eventID and fields as a BinXML event record
// and appends it to the pending buffer.
// fields keys must match the CEPA data field names.
func (w *Writer) WriteRecord(eventID int, fields map[string]string) error { ... }

// Close flushes all pending records to disk and finalizes the .evtx file.
func (w *Writer) Close() error { ... }
```

### Pattern 2: cee-exporter BinaryEvtxWriter Adapter

**What:** `BinaryEvtxWriter` in cee-exporter becomes a thin adapter that translates `WindowsEvent` to `map[string]string` and delegates to `go-evtx`.
**When to use:** Maintains the existing `evtx.Writer` interface contract within cee-exporter while delegating all format logic to go-evtx.

```go
// Source: adapter pattern — writer_evtx_notwindows.go post-extraction

//go:build !windows

package evtx

import (
    "context"
    goevtx "github.com/fjacquet/go-evtx"
)

type BinaryEvtxWriter struct {
    w *goevtx.Writer
}

func NewBinaryEvtxWriter(evtxPath string) (*BinaryEvtxWriter, error) {
    w, err := goevtx.New(evtxPath)
    if err != nil {
        return nil, err
    }
    return &BinaryEvtxWriter{w: w}, nil
}

func (b *BinaryEvtxWriter) WriteEvent(_ context.Context, e WindowsEvent) error {
    fields := windowsEventToFields(e)
    return b.w.WriteRecord(e.EventID, fields)
}

func (b *BinaryEvtxWriter) Close() error {
    return b.w.Close()
}

// windowsEventToFields translates a WindowsEvent to the map[string]string
// expected by go-evtx's WriteRecord. Keys match CEPA/Security audit data fields.
func windowsEventToFields(e WindowsEvent) map[string]string {
    return map[string]string{
        "SubjectUserSid":    e.SubjectUserSID,
        "SubjectUserName":   e.SubjectUsername,
        "SubjectDomainName": e.SubjectDomain,
        "SubjectLogonId":    e.SubjectLogonID,
        "ObjectServer":      "Security",
        "ObjectType":        e.ObjectType,
        "ObjectName":        e.ObjectName,
        "HandleId":          e.HandleID,
        "AccessList":        e.Accesses,
        "AccessMask":        e.AccessMask,
        "ProcessId":         "",
        "ProcessName":       "",
    }
}
```

### Pattern 3: WriteRaw Implementation

**What:** WriteRaw accepts a complete pre-encoded record byte slice (already wrapped with `wrapEventRecord`) and appends it.
**Note:** WriteRaw and WriteRecord share the same underlying buffer — callers should use one or the other, not both, to avoid record ID sequencing issues.

```go
// WriteRaw appends a pre-encoded event record (must include record header) to the buffer.
// The caller is responsible for constructing a valid record via wrapEventRecord.
func (w *Writer) WriteRaw(chunk []byte) error {
    w.mu.Lock()
    defer w.mu.Unlock()
    w.records = append(w.records, chunk...)
    w.recordID++
    return nil
}
```

### Pattern 5: pkg.go.dev Publishing

**What:** Publishing go-evtx so `go get github.com/fjacquet/go-evtx` works from any machine.
**Steps:**


### Anti-Patterns to Avoid

- **Importing cee-exporter types in go-evtx:** go-evtx must never import `github.com/fjacquet/cee-exporter`. The `WindowsEvent` struct stays in cee-exporter; the adapter translates it before calling go-evtx.
- **Changing EVTX format logic during extraction:** Phase 8.5 is a pure extraction. The `flushChunkLocked()` stub stays as-is; fixing it is Phase 10's job. Changing format logic during extraction conflates two concerns and risks breaking the verified python-evtx output.
- **Using a v1.0.0 tag immediately:** Start with v0.1.0. v0.x semantic versioning allows breaking API changes without a v2 module path. The API will evolve in Phases 9–11.
- **Moving `WindowsEvent` to go-evtx:** Premature coupling. The `map[string]string` API at the go-evtx boundary is intentionally generic; it keeps go-evtx useful as a standalone forensics library.
- **Putting build tags (`//go:build !windows`) in go-evtx:** go-evtx is platform-agnostic pure Go. Build tags belong only in cee-exporter.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CRC32 computation | Custom CRC | `hash/crc32` stdlib | Already used and verified in evtx_binformat.go |
| UTF-16LE encoding | Custom encoder | `unicode/utf16` stdlib | Already used in evtx_binformat.go and writer_evtx_notwindows.go |
| LE binary serialization | Manual byte shifting | `encoding/binary` stdlib | Already used throughout; readable and correct |
| Module versioning | Custom version scheme | semantic versioning + git tags | Go proxy requires semver tags; deviating breaks `go get` |
| EVTX format correctness oracle | Re-implementing parser | python-evtx | Already validated in current codebase; use the same validation approach in go-evtx CI |

**Key insight:** The extraction moves proven, tested code. The risk is in the adapter wiring and import paths, not in the format logic itself. Do not improve the format logic during extraction.

## Common Pitfalls

### Pitfall 1: Circular dependency — go-evtx imports cee-exporter
**What goes wrong:** If `go-evtx` imports `github.com/fjacquet/cee-exporter` to access `WindowsEvent`, you have a circular dependency that the Go toolchain rejects.
**Why it happens:** `collectSubstitutions` currently takes a `WindowsEvent` — copy-paste migration drags in the type.
**How to avoid:** go-evtx receives only `map[string]string`; the adapter in cee-exporter does the translation.
**Warning signs:** `go mod tidy` in cee-exporter complains about an import cycle.

### Pitfall 2: Test file build tags not updated
**What goes wrong:** `evtx_binformat_test.go` has no build tag (correct). `writer_evtx_notwindows_test.go` has `//go:build !windows`. When ported to go-evtx, the `!windows` tag is unnecessary (go-evtx is platform-agnostic) but harmless. The risk is forgetting to remove the tag and causing tests to be skipped on unexpected platforms.
**How to avoid:** Remove all build tags from test files in go-evtx. go-evtx is pure Go with no OS-specific code.
**Warning signs:** `go test ./...` on Windows skips tests.

### Pitfall 3: Internal package access — test package name mismatch
**What goes wrong:** Current tests use `package evtx` (white-box, same package as code). If go-evtx uses `package evtx` and the test files are ported verbatim, this works correctly. If the package name changes, unexported symbols become inaccessible.
**How to avoid:** Use `package evtx` as the package name in go-evtx (matching the current codebase). Keep test files in `package evtx` for white-box access.
**Warning signs:** `undefined: buildBinXML` or similar unexported symbol errors in tests.

### Pitfall 4: recordID sequencing broken by WriteRaw + WriteRecord mix
**What goes wrong:** WriteRaw and WriteRecord share the same `recordID` counter and `records` buffer. If a caller uses both in the same session, record IDs may be non-contiguous or the chunk header fields may be wrong.
**How to avoid:** Document in the go-evtx README that WriteRaw and WriteRecord are mutually exclusive within a single Writer session. Do not enforce this in code (it adds complexity); document the contract clearly.
**Warning signs:** python-evtx reports non-sequential record IDs.

### Pitfall 5: pkg.go.dev not indexing due to private repository
**What goes wrong:** If the GitHub repository is private when the tag is pushed, proxy.golang.org cannot crawl it. `go get github.com/fjacquet/go-evtx` fails with a 404.
**How to avoid:** Ensure the repository is public before pushing the first semver tag. The GOPROXY crawl step confirms indexing.
**Warning signs:** `go get github.com/fjacquet/go-evtx@v0.1.0` returns `not found` from proxy.golang.org.

### Pitfall 6: cee-exporter tests fail due to missing go.sum entry
**What goes wrong:** After `go get github.com/fjacquet/go-evtx@v0.1.0`, `go.sum` is updated automatically by `go mod tidy`. If `go.sum` is not committed, CI fails with checksum mismatch.
**How to avoid:** Commit both `go.mod` and `go.sum` changes as part of the EXT-05 task. Run `go mod tidy` before committing.
**Warning signs:** CI fails with "missing go.sum entry for module".

### Pitfall 7: binXMLChunkOffset calculation broken by package move
**What goes wrong:** `buildBinXML` receives a `binXMLChunkOffset` that is the chunk-relative byte offset of the BinXML payload. This value is computed in `WriteEvent` as `evtxRecordsStart + uint32(len(w.records)) + evtxRecordHeaderSize`. All three constants (`evtxRecordsStart`, `evtxRecordHeaderSize`) must be in scope in go-evtx. If any constant is accidentally left in cee-exporter, the BinXML offset calculation produces corrupt NameNode offsets.
**How to avoid:** All constants from `evtx_binformat.go` and the const block in `writer_evtx_notwindows.go` must move together to go-evtx. Run python-evtx validation after extraction as the correctness oracle.
**Warning signs:** python-evtx fails to parse the generated .evtx file or produces garbled element names.

Verified patterns from the existing codebase:

### go-evtx Writer construction (mirrors current NewBinaryEvtxWriter)
```go
// Source: cee-exporter/pkg/evtx/writer_evtx_notwindows.go — NewBinaryEvtxWriter
func New(path string) (*Writer, error) {
    if path == "" {
        return nil, fmt.Errorf("evtx: path must be non-empty")
    }
    if err := os.MkdirAll(filepath.Dir(path), 0o755); err != nil {
        return nil, fmt.Errorf("evtx: create parent directory: %w", err)
    }
    return &Writer{
        path:     path,
        recordID: 1,
        firstID:  1,
    }, nil
}
```

### WriteRecord (adapter over buildBinXML)
```go
// Source: adapted from WriteEvent in writer_evtx_notwindows.go
func (w *Writer) WriteRecord(eventID int, fields map[string]string) error {
    w.mu.Lock()
    defer w.mu.Unlock()

    binXMLChunkOffset := evtxRecordsStart + uint32(len(w.records)) + evtxRecordHeaderSize
    payload := buildBinXML(eventID, fields, binXMLChunkOffset)
    // timestamp: accept as parameter or use time.Now() — decision for go-evtx API
    ts := toFILETIME(time.Now())
    rec := wrapEventRecord(w.recordID, ts, payload)
    w.records = append(w.records, rec...)
    w.recordID++
    return nil
}
```

**Note on timestamp:** `buildBinXML` currently gets the timestamp from `WindowsEvent.TimeCreated`. In go-evtx's `WriteRecord(eventID int, fields map[string]string)` signature, the timestamp must be either passed explicitly or included in `fields`. Recommended: add an optional `"TimeCreated"` key to fields (RFC3339 string) with fallback to `time.Now()`. This preserves the CEPA event timestamp in the output.

## Migration Path: Keeping cee-exporter Tests Green

The extraction must be done in a sequence that keeps `make test` passing at every step.

### Recommended Wave Order

**Wave 1 — Create go-evtx repo and copy code (cee-exporter unchanged)**
1. Create `github.com/fjacquet/go-evtx` GitHub repository (public).
2. Copy `evtx_binformat.go` → `go-evtx/binformat.go`; copy `evtx_binformat_test.go` → `go-evtx/binformat_test.go`.
3. Copy BinXML encoding functions from `writer_evtx_notwindows.go` into `go-evtx/binxml.go`.
4. Define `Writer` struct and `New`/`WriteRaw`/`WriteRecord`/`Close` in `go-evtx/evtx.go`.
5. Port the writer tests from `writer_evtx_notwindows_test.go` to `go-evtx/evtx_test.go` (adapting for the new API).
6. Confirm `go test ./...` passes in go-evtx.
7. Tag `v0.1.0` and push; confirm pkg.go.dev indexes.

**Wave 2 — Update cee-exporter to use go-evtx (EXT-05)**
1. `go get github.com/fjacquet/go-evtx@v0.1.0` in cee-exporter.
2. Replace the body of `writer_evtx_notwindows.go` with the thin adapter.
3. Remove the extracted functions from `writer_evtx_notwindows.go` (they now live in go-evtx).
4. Keep `evtx_binformat.go` and `evtx_binformat_test.go` in cee-exporter or remove them — since their contents moved to go-evtx, remove them and update any remaining callers.
5. Run `make test` — all existing tests must pass.
6. The writer tests in `writer_evtx_notwindows_test.go` continue to test the adapter's behaviour; they do not need to be deleted.

**Critical invariant:** At the end of Wave 2, `make test` in cee-exporter runs the same writer tests as before. The tests now exercise the adapter → go-evtx path, which is correct.

## Open Questions

1. **Timestamp in WriteRecord signature**
   - What we know: `buildBinXML` currently receives a `WindowsEvent` with `TimeCreated time.Time`; go-evtx's `WriteRecord(eventID int, fields map[string]string)` must carry the timestamp somehow.
   - What's unclear: Should timestamp be a required third parameter `WriteRecord(eventID int, ts time.Time, fields map[string]string)` or a reserved key in `fields` (`"TimeCreated"`)?
   - Recommendation: Use `"TimeCreated"` as a reserved key in fields (RFC3339Nano format). This keeps the signature simple and avoids a future breaking change. The adapter sets it from `WindowsEvent.TimeCreated`.

2. **ProviderName and Computer fields**
   - What we know: `collectSubstitutions` uses `e.ProviderName` and `e.Computer` as substitution subs 0 and 4.
   - What's unclear: These are metadata fields, not data fields. They could be WriteRecord parameters or special reserved keys.
   - Recommendation: Use reserved keys `"ProviderName"` and `"Computer"` in the `fields` map. The adapter populates them from `WindowsEvent.ProviderName` and `WindowsEvent.Computer`.

3. **WriteRaw record ID management**
   - What we know: WriteRaw must increment `w.recordID` to keep sequence numbers contiguous. But the caller provides a pre-encoded record that already has a recordID baked in (via `wrapEventRecord`).
   - What's unclear: Does WriteRaw take a raw record (already wrapped) or a raw BinXML payload (before wrapping)?
   - Recommendation: WriteRaw takes a raw BinXML payload (not yet wrapped). go-evtx wraps it with `wrapEventRecord(w.recordID, ...)` internally. This maintains correct record ID sequencing. Document this clearly in the README.
